## Iteration 2 — Content pipeline and components

Date: 2026-06-24.

### Scope read (read-first, exhaustive)

- `src/content.config.ts` (collection schema).
- `src/plugins/rehype-reveal-sections.mjs` (plain-Markdown sectioning plugin).
- All 5 components: `Slide`, `HudSlide`, `KvCard`, `Chat`, `Headers`.
- All 3 decks: `hackergpt-osint/index.mdx`, `hardest-workers/index.mdx`,
  `work-smarter-checklist/index.md`.

### Method note (adversarial review attempted)

A parallel adversarial review workflow (3 dimension reviewers + per-finding verification) was
launched first, but all 3 reviewers died on a transient server-side API 529 (Overloaded) — so
its `0 findings` result is meaningless (the reviewers never ran). Rather than hammer an
overloaded API, this iteration used direct read-first analysis and verified render-neutrality
empirically by diffing the built HTML (per-file SHA-1 before/after), which is stronger evidence
than a reviewer's opinion on render-neutrality.

### Findings

1. **Spanish comments across the content-pipeline code files.** `content.config.ts`,
   `rehype-reveal-sections.mjs`, and the 5 components carried Spanish inline/JSDoc comments.
   Conflicts with the user's stated convention (English for all code and docs).
2. **Defensive-guard asymmetry in the plugin.** `extractFrame` guards
   `node.children && node.children[0]` (`rehype-reveal-sections.mjs:69`) but `toNotesIfNeeded`
   accessed `node.children[0]` unguarded (`:86`). Not a live bug (hast element nodes always have
   a `children` array), but an inconsistency worth aligning.
3. **MD vs MDX coherence: healthy.** 2 MDX decks author their own `<section>` via components
   (`Slide`/`HudSlide`) and the plugin correctly skips them via the `isAuthored` check
   (`rehype-reveal-sections.mjs:18-23`); 1 plain-MD deck is sectioned by the plugin. Schema is
   solid: `theme` is build-time validated against the registry (`content.config.ts:21`), `cover`
   uses `image()`, defaults are sane. Component props are fully typed (tuple types in `KvCard`,
   a `Message` interface in `Chat`).
4. **Speaker-notes `<aside>` duplication.** The block `{note && <aside class="notes"
   set:html={note} />}` is identical in `Slide.astro:37` and `HudSlide.astro:32`.
5. **`set:html` on speaker notes** (`Slide.astro:37`, `HudSlide.astro:32`). Renders note text
   as HTML. Intentional for a static, single-author site (lets notes carry formatting); not a
   security issue here.
6. **Deep relative imports in MDX decks** (`../../../components/...`). A DX wart.
7. **`category` default `'Sin categoria'`** (`content.config.ts:14`). A Spanish, UI-facing
   fallback label that is currently dead (all 3 decks set `category`).
8. **Plugin edge case (theoretical).** An `.mdx` deck containing only imports/exports + plain
   Markdown separated by `---` (no JSX component, no `<section>`) would be classified as
   non-authored and transformed, and its `mdxjsEsm` import/export nodes would land inside the
   first `<section>`. Not a defect for any current deck (every MDX deck uses components).

### Changes applied (safe and reversible; render-neutral)

- **English comments** in `content.config.ts`, `rehype-reveal-sections.mjs`, and the 5
  components (`Slide`, `HudSlide`, `KvCard`, `Chat`, `Headers`). Comment-only (finding 1).
- **Guard alignment** in `rehype-reveal-sections.mjs:86`: `node.children[0]` ->
  `node.children && node.children[0]`, matching `extractFrame` (finding 2). Strictly more
  defensive; never changes output for valid hast input.

Render-neutrality proof: `pnpm build` green, and the SHA-1 of each of the 4 built HTML files is
identical before and after the changes (`hackergpt-osint`, `hardest-workers`,
`work-smarter-checklist`, `index`). Only non-comment code change in tracked files is the guard
line. (The 4 HUD components are still untracked WIP, so their comment edits do not appear in
`git diff`; the 226-line `index.md` deletion in `git diff` is the user's pre-existing md->mdx
migration, untouched here.)

### Deferred changes / recommendations

- **Do NOT extract a shared `SpeakerNotes` component (finding 4)** for now: 2 call sites of a
  one-line block is below the bar for extraction (over-engineering). Revisit if a third
  notes-bearing slide type appears or notes need shared logic (e.g. markdown-in-notes).
- **Import path alias (finding 6).** Add a `@components/*` path in `tsconfig.json` to replace
  `../../../components/...`. Deferred to iteration 4 (config/DX dimension).
- **`category` default (finding 7).** Left untouched: it is content/UI-facing Spanish text for a
  Spanish-content site, and changing it is a UI-string decision, not a code-comment one. It is
  currently dead (every deck sets `category`). Open question for the user: anglicize to
  `'Uncategorized'`, or keep Spanish to match deck content?
- **Document the authoring contract (finding 8).** Either note in the plugin that MDX decks must
  use components/`<section>`, or defensively drop `mdxjsEsm` nodes before sectioning. Not applied
  (speculative; would touch behavior for a pattern no deck uses).
- **Cross-cutting: remaining Spanish comments** elsewhere in the repo (layouts, pages, themes)
  will be anglicized as their dimensions come up (iterations 3-4) and reconciled in iteration 5.

- After changes: `pnpm build` OK — 4 pages.

## Iteration 3 — Theme system and CSS

Date: 2026-06-24.

### Scope read (read-first, exhaustive)

- `src/themes/themes.css` (aggregator), `src/themes/registry.ts` (single source of truth).
- `src/themes/_base.css` (180 lines: structural layout + semantic-token mapping, `@layer`).
- `src/themes/{steel-light,midnight}/theme.css` (16 lines each: token-only overrides).
- `src/themes/hud/theme.css` (1587 lines: the full terminal aesthetic).
- How themes are wired: `RevealLayout.astro` imports `themes.css` once; the active theme is
  selected by `:root[data-theme]`. All theme CSS bundles into one minified file
  (`dist/_astro/_..*.css`).

### Headline finding: ~1229 lines of dead CSS in the hud theme

The `hud` theme defined a huge family of frame-experiment classes — `.border-hud`,
`.border-panel`, `.border-right`, `.border-hud-l/-b/-180`, `.border-left`, `.border-mix-*`
(ac/d/e/180, -cr/-cl), `.border-ring-dnotch(-180/-x/-y)`, `.border-dnotch`, `.border-rdframe`,
`.border-notch`, `.border-xhair`, `.border-reticle`, `.border-frametk`,
`.border-cyber/deusex/elite/cut-diag`, and a full `.border-dc1`..`.border-dc20+` series
(`dc` = design-comparison). These are leftovers from the frame-design exploration (the same
exploration the deleted root PNGs in iteration 1 documented). A deterministic class-usage scan
proved NONE of them appear in any deck, component, layout, or built HTML. The frames the decks
actually use are the `.fr-*` family: `fr-bracket`, `fr-cyber`, `fr-deusex`, `fr-elite`,
`fr-notch`, `fr-placa` (`hud/theme.css:334-358`), applied via `HudSlide frame=`.

Embedded inside the dead range were three LIVE rules that had to be preserved: the default
frame's notch helpers `.slides > section > .frame-notch` / `.frame-notch-r` and the default
frame's border ring `.slides > section::after` (a live island originally at lines 1322-1360).

### Other findings

1. **Spanish comments across the whole theme system** (all 6 files) — conflicts with the
   English convention.
2. **`_base.css` doc inaccuracy** (original `_base.css:7`): the comment said themes define via
   `.reveal[data-theme="..."]`, but they actually use `:root[data-theme="..."]` (the attribute
   is on `<html>`). Corrected in the English rewrite.
3. **Duplicated font tokens.** `midnight` and `steel-light` define identical `--font`/
   `--font-head` (Manrope/Quicksand); only `hud` differs (JetBrains Mono).
4. **Token coverage is consistent** across all 3 themes (`--bg-a/-b`, `--shade`, `--ink`,
   `--ink-dim`, `--accent`, `--glow`, `--radius`, `--font`, `--font-head`). `hud` sets
   `--radius: 8px` vs `44px` for the soft themes — intentional (sharp terminal vs soft cards).

### Changes applied (safe and reversible; proven render-neutral)

- **Removed the entire dead `.border-*` family** from `hud/theme.css`: 1587 -> 358 lines
  (-1229). Kept the live `.fr-*` frames, the `frame-notch` island, chat/kv-card/ep-card blocks,
  the base section frame, tokens, and terminal identity.
- **Translated all theme-system comments to English** (`themes.css`, `registry.ts`, `_base.css`,
  `midnight`, `steel-light`, and the surviving `hud` comments). Comment-only.
- **Corrected the `_base.css` selector doc note** (`.reveal[data-theme]` -> `:root[data-theme]`).

Render-neutrality proofs (two independent oracles):
- Deletion: normalized the minified CSS bundle into per-rule lines and diffed before/after. 132
  rules removed, EVERY one a `.border-*` rule; 0 rules added; every surviving rule byte-identical.
  Since the removed selectors match zero elements in any rendered deck, the computed styles are
  unchanged.
- Comments: after all comment edits, the rebuilt bundle is byte-identical to the post-deletion
  bundle (same content hash, 84414 bytes) — comments are stripped at build, so a byte-identical
  bundle proves no declaration changed.
- HTML render counts unchanged: `hackergpt-osint` 13/13/8kv/2chat/1ep (6 `fr-*` frames),
  `hardest-workers` 4/4, `work-smarter-checklist` 4/2, `index` 3/0.
- CSS bundle shrank 215726 -> 84414 bytes (-61%).

### Deferred changes / recommendations

- **Hoist shared font tokens (finding 3).** Define `--font`/`--font-head` defaults once (e.g. in
  `_base.css` under the `base` layer) and let `hud` override under the `theme` layer; drop the
  duplicate from `midnight`/`steel-light`. Render-neutral if the layer order is respected, but a
  token-cascade change warrants its own verification — deferred to keep this diff focused.
- **Optional hud modularization.** At 358 lines `hud/theme.css` is now manageable, but it still
  mixes tokens / terminal identity / chat / kv-card / ep-card / base frame / `fr-*` frames. If it
  grows, split into `hud/{tokens,identity,cards,frames}.css` imported in order by a barrel. Not
  applied (the file is small enough now; splitting only reorganizes).

### Build status

- Before changes: `pnpm build` OK — CSS bundle 215726 bytes.
- After changes: `pnpm build` OK — CSS bundle 84414 bytes (-61%); `hud/theme.css` 1587 -> 358.
- Render-neutral, proven by rule-level bundle diff (deletion) and byte-identical bundle
  (comments). HTML counts unchanged.
